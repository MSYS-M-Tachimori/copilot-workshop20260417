# Pomodoro Timer 段階的実装計画

このドキュメントは、ポモドーロタイマーアプリケーションを段階的に実装するための計画をまとめたものです。

## 1. 実装粒度の基本方針
1. 1フェーズは「1〜2日」「1PR」「1つの完了条件」を目安にする
2. 各フェーズで必ず「動く画面」または「叩けるAPI」のどちらかを完成させる
3. ドメインロジックは純粋関数優先、Flaskルートは配線中心に保つ
4. テストは同時実装し、後追いでまとめて書かない

## 2. 段階的実装計画

### フェーズ0: 土台構築
- 目的: 実装を増やしても崩れない最小構成を作る
- 実装:
  - Flask起動
  - テンプレート配信
  - 静的ファイル配信
  - 推奨ディレクトリ骨格作成
  - 開発手順のREADME反映
- 完了条件:
  - トップ画面が表示され、CSS/JSが読み込まれる
- テスト:
  - Flaskの最小ヘルスチェック相当テスト

### フェーズ1: UIモック忠実再現（静的）
- 目的: 添付画像の見た目を先に完成させる
- 実装:
  - タイトル
  - 円形プログレス
  - 残り時間表示
  - 開始/リセットボタン
  - 今日の進捗カード
- 完了条件:
  - デザインがPC/モバイルで崩れず表示される
- テスト:
  - 画面表示の手動確認チェックリスト

### フェーズ2: フロント状態管理（ローカルのみ）
- 目的: UIに状態遷移を与える
- 実装:
  - state/reducer導入
  - START/PAUSE/RESET/COMPLETE/SWITCH_MODE実装
  - 不正遷移ガード
  - render分離
- 完了条件:
  - API未接続でも状態遷移が正しく動く
- テスト:
  - reducerのユニットテスト

### フェーズ3: 高精度タイマー実装
- 目的: 実運用でズレにくいタイマーを作る
- 実装:
  - 終了予定時刻ベースで残時間を算出
  - タブ復帰時の再計算
  - 0秒到達でCOMPLETE遷移
- 完了条件:
  - バックグラウンド復帰後も大きな時刻ズレが出ない
- テスト:
  - 時間計算純粋関数のユニットテスト
  - 境界値テスト（0秒、直前、超過）

### フェーズ4: セッション完了API（保存なし暫定）
- 目的: クライアントとサーバーの契約を先に固定する
- 実装:
  - POST /api/sessions/complete
  - 入力検証
  - レスポンス形式確定
  - UseCase呼び出し配線
  - 保存先は一旦InMemoryでも可
- 完了条件:
  - 完了時にAPIへ正しいペイロードを送信できる
- テスト:
  - API正常系
  - バリデーション異常系

### フェーズ5: SQLite永続化 + 今日の進捗API
- 目的: 記録と集計を本番形にする
- 実装:
  - DB初期化
  - Repository実装
  - complete保存
  - GET /api/progress/today
  - 日次集計ロジック
- 完了条件:
  - 画面再読み込み後も進捗値が保持される
- テスト:
  - Repository結合テスト
  - 日次集計テスト
  - API結合テスト

### フェーズ6: アプリケーション層整理とテスト強化
- 目的: 将来拡張しやすい構造に固める
- 実装:
  - UseCase分離の徹底
  - Clock抽象化
  - 必要に応じた値オブジェクト導入
  - Flaskルートの薄化
- 完了条件:
  - Flask依存なしで主要ビジネスロジックをテスト可能
- テスト:
  - domain/applicationユニットテスト強化
  - 時刻境界テスト（00:00跨ぎ）

### フェーズ7: UX仕上げと拡張入口
- 目的: MVPを使える品質に仕上げる
- 実装:
  - エラーメッセージ表示
  - ローディング表示
  - 操作不能時の制御
  - 通知・設定機能の拡張ポイント準備
- 完了条件:
  - 主要ユーザーフローで操作が詰まらない
- テスト:
  - 手動E2Eシナリオ（開始→完了→保存→再表示）

## 3. チケット粒度の目安
1. 1チケットは「1成果物 + 1テスト観点」で分割する
2. 200〜500行程度の変更に収める
3. 依存が強い作業はUI/ロジック/API契約/永続化で分割する

## 4. 短期スプリント例
1. Sprint A: フェーズ0〜1（見た目完成）
2. Sprint B: フェーズ2〜3（ローカルでタイマー完成）
3. Sprint C: フェーズ4〜5（記録と進捗の永続化）
